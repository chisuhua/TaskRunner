# Phase 1.5 Stretch — Fix CudaScheduler dynamic_cast<CudaStub*> Hardcoding

> **Status**: PROPOSED · **Effort**: 0.5-1 day · **Priority**: P1 (unblocks cross-repo real backend path)

## Executive Summary

`src/test_fixture/cuda_scheduler.cpp` contains 5 `dynamic_cast<CudaStub*>(driver_)` sites that hard-bind the CudaScheduler to the CudaStub implementation. This prevents GpuDriverClient (the System C ioctl-based real backend living in UsrLinuxEmu) from serving `submitMemAlloc`, `submitMemFree`, `submitMemcpyH2D`, `submitMemcpyD2H`, and `submitLaunch` — blocking the end-to-end path:

```
libcuda_taskrunner.so → CudaRuntimeApi → CudaScheduler → IGpuDriver → GpuDriverClient → UsrLinuxEmu
```

H-3.5 already resolved 3 out of 8 dynamic_cast sites (set_stub_mode, initialize, shutdown) by adding non-pure virtual methods to IGpuDriver. The remaining 5 require routing CUDA Driver API semantics through the IGpuDriver virtual interface.

## Problem Statement

### Current Architecture (Broken)

```
CudaScheduler
  │
  ├── driver_->alloc_bo()          // ✅ IGpuDriver virtual — works with any backend
  ├── driver_->submit_memcpy()     // ✅ IGpuDriver virtual — works with any backend
  ├── driver_->submit_launch()     // ✅ IGpuDriver virtual — works with any backend
  │
  └── dynamic_cast<CudaStub*>(driver_)
        └── stub->mem_alloc()      // ❌ CudaStub-specific — GpuDriverClient returns -ENOSYS
        └── stub->mem_free()       // ❌ " "
        └── stub->memcpy_h2d()     // ❌ " "
        └── stub->memcpy_d2h()     // ❌ " "
        └── stub->launch_kernel()  // ❌ " "
```

### Target Architecture (After Fix)

```
CudaScheduler
  │
  ├── driver_->alloc_bo()          // ✅ IGpuDriver virtual
  ├── driver_->alloc_bo_vram()     // ✅ IGpuDriver virtual
  ├── driver_->free_bo()           // ✅ IGpuDriver virtual
  ├── driver_->submit_memcpy()     // ✅ IGpuDriver virtual
  ├── driver_->submit_launch()     // ✅ IGpuDriver virtual
  └── (no dynamic_cast needed)
```

## The 5 Sites

| # | Method | Line | Current Call | Replacement | IGpuDriver Method |
|---|--------|------|-------------|-------------|-------------------|
| 1 | `submit_mem_alloc` | 99-104 | `stub->mem_alloc(size, &device_ptr)` | `driver_->alloc_bo_vram(size, GPU_BO_DEVICE_LOCAL)` | `alloc_bo_vram` |
| 2 | `submit_mem_free` | 145-148 | `stub->mem_free(device_ptr)` | `driver_->free_bo(bo_handle)` | `free_bo` |
| 3 | `submit_memcpy_h2d` | 186-189 | `stub->memcpy_h2d(...)` | `driver_->submit_memcpy(stream, src, dst, size, true)` | `submit_memcpy` |
| 4 | `submit_memcpy_d2h` | 225-228 | `stub->memcpy_d2h(...)` | `driver_->submit_memcpy(stream, src, dst, size, false)` | `submit_memcpy` |
| 5 | `submit_launch` | 267-277 | `stub->launch_kernel(params, &task_id)` | `driver_->submit_launch(stream, kernel_index, grid, block)` | `submit_launch` |

### Key Design Decision: BO Handle vs Device Pointer Mapping

`alloc_bo()`/`alloc_bo_vram()` return `uint64_t bo_handle`, but CudaScheduler's MemoryManager tracks `device_ptr`. The fix must add a `bo_handle` field to MemoryDescriptor and maintain the mapping:

```cpp
// MemoryDescriptor 扩展
struct MemoryDescriptor {
    uint64_t device_ptr;   // 已有
    uint64_t bo_handle;    // 新增: IGpuDriver::alloc_bo 返回值
    size_t size;
    // ...
};
```

## Non-Goals

- Do NOT add CUDA Driver API methods (mem_alloc, memcpy_h2d, etc.) to IGpuDriver — these are CUDA-layer semantics, not IOCTL-layer
- Do NOT change CudaStub's existing CUDA Driver API methods — they remain for backward compatibility
- Do NOT break existing 76/76 test suite
- Do NOT change the UMD shim layer (libcuda_taskrunner.so)
